## Design Context

### Users

**Primary workspace users (all-day):** Project Managers, Team Leads, QA Leads managing release pipelines for teams of 15–100+ people. They plan resources, monitor blockers, and track the "big picture" across 500+ tasks. This is their professional command center.

**Occasional users:** Developers — open once per sprint/release cycle to check the global plan or grab a Slack channel link. They live in IDEs, Git, and Jira; this tool must respect that and get out of their way.

**Core job to be done:** Answer "what is blocked, why, and whose court is the ball in?" — without clicking into individual task cards. The pipeline is the main artifact.

**Scale requirements:** Must remain readable and usable with 500+ tasks. Performance and information density are non-negotiable.

### Brand Personality

**3 words:** Structured. Automated. Transparent.

The tool feels like a precision instrument — a well-calibrated industrial engine that pulls data from everywhere (Jira, Git, Confluence, Slack) and surfaces it without noise or ceremony. It has opinions. It does not apologize.

**Must NOT feel like:** Playful (no confetti, no friendly illustrations), chaotic, slow, bloated legacy enterprise (no MS Project 2010 energy), visually noisy (no Jira-style menu sprawl).

**Emotional register:** The calm confidence of a control room. Not sterile — it has character. But it never shouts.

### Aesthetic Direction

**Theme:** Dark by default (developers, engineers, and QA work at night before releases; dark mode matches their other tools — IDE, terminal, Git UI). Reports/exports must support a high-contrast light variant for stakeholders and print.

**References:**
- **Linear** — speed, precision, keyboard-first, excellent dark mode, no visual noise
- **Notion timeline/database views** — clean, structured, data-logic without clutter
- **Vercel dashboard** — strict, functional, data-focused; respects the engineer's time
- **Datadog** — information density done right

**Anti-references:** Jira (overwhelming menus), Trello (informal), MS Project (bloated legacy), anything with confetti or excessive animation.

**Color palette:** Deep graphite base with warm amber accent. The current cyan-gradient-on-dark palette is the AI design cliché — replace entirely. Amber reads as "precision instrument": measurement scales, engineering diagrams, warning indicators. It is distinctive without being loud.

```
Background:   oklch(0.12 0.010 60)   — deep graphite, warm-tinted
Surface:      oklch(0.16 0.008 60)   — card/panel surface
Elevated:     oklch(0.20 0.007 60)   — popovers, dropdowns
Border:       oklch(0.24 0.006 60)   — subtle dividers
Text primary: oklch(0.94 0.005 60)   — near-white, warm
Text muted:   oklch(0.55 0.008 60)   — secondary labels
Accent:       oklch(0.75 0.14 85)    — warm amber — primary actions, active states
Accent dim:   oklch(0.75 0.14 85 / 0.15) — amber ghost for hover surfaces
```

**Task state palette** (pipeline stages, distinct and meaningful):
```
NotStarted:   oklch(0.40 0.006 60)   — neutral graphite
InDev:        oklch(0.75 0.14 85)    — amber (active work)
MrToRelease:  oklch(0.72 0.10 190)   — teal (in transit)
InTest:       oklch(0.68 0.12 250)   — slate-blue (under review)
MrToMaster:   oklch(0.70 0.10 300)   — muted violet (final gate)
Completed:    oklch(0.65 0.10 145)   — muted green (done)
```

**Integration status dot colors** (inline micro-indicators):
```
Blocked/waiting: oklch(0.75 0.14 85)   — amber = waiting on something
Passed/clear:    oklch(0.65 0.10 145)  — green = no action needed
Failed/error:    oklch(0.65 0.14 25)   — muted red = broken
```

**Typography:** Cyrillic-first font selection (the UI has Russian text).

- **Display / headings:** [Geologica](https://fonts.google.com/specimen/Geologica) — geometric variable font with full Cyrillic support. Use the condensed axis (wdth 75–85) for Gantt bar labels and tight spaces. Crisp, engineered, zero warmth — exactly right for a control panel header.
- **Body / UI:** [Onest](https://fonts.google.com/specimen/Onest) — clean variable grotesque designed with Cyrillic as a first-class script. Has tabular numbers (critical for task IDs and counts). Feels engineered, not friendly-startup.
- **IDs / code / branch names:** Geist Mono or system monospace — Jira IDs (PROJ-1234) and branch names always in mono to visually separate identifiers from natural language.

**Layout philosophy:** Left-aligned, asymmetric, information-dense. No decorative cards wrapping everything. No centered hero moments. The grid is a tool for organization, not decoration.

### Design Principles

1. **Status at a glance.** Integration blockers (Confluence approval pending, Git MR open, Jira ticket stalled) are visible inline on the task bar — micro-indicators with logo + color dot. No drilling in just to see whose court the ball is in.

2. **Data density over decoration.** Every visual element encodes information. Spacing conveys hierarchy. Color encodes state. Size conveys importance. Nothing decorates for its own sake.

3. **Calm precision.** The system handles extraordinary complexity quietly. It does not alarm, animate unnecessarily, or use motion to fill silence. When something moves, it means something changed.

4. **Keyboard-first fluency.** PMs who live in this tool all day should be able to navigate, filter, and act without reaching for the mouse. Tab stops, keyboard shortcuts, and command palette are first-class.

5. **Scalable clarity.** The interface must remain readable and decisive at 500+ tasks. Hierarchy is structural, not decorative — it holds up under load.

### Notes for Implementation

- Replace the cyan-to-green gradient primary button immediately. Use solid amber (`oklch(0.75 0.14 85)`) with dark text — no gradients on interactive elements.
- The glassmorphism backdrop-filter on the header can stay if subtle, but remove it from card surfaces — it reads as decorative noise at scale.
- Jira IDs should always render in Geist Mono / system mono to visually distinguish identifiers from prose.
- The role badges (Developer / Manager) should use state palette colors, not the current cyan/purple — they encode meaning (role = a type of state), not brand flavor.
- For the integration blocker indicators: 16×16px logo icon (Confluence, GitHub, Jira, Slack) + 6px dot in the top-right corner. Color = integration status. Tooltip on hover explains the specific blocker.
- The "jump to Slack channel" quick action should appear on task row hover — not always visible, revealed through progressive disclosure.

### CSS Architecture

Parallel design tasks must not collide. Enforce that by splitting styles by concern and colocating with the component that owns them.

**Layer map:**

```
src/styles/
├── tokens.css              ← OKLCH vars, shadows, state palette  (owner: /colorize)
├── typography.css          ← font-family, h1-h6 defaults         (owner: /typeset)
├── reset.css               ← a, body, button defaults            (rare-edit)
└── primitives/
    ├── button.css          ← .primary-button, .secondary-button
    ├── field.css           ← .field, .field__input, .error-text
    ├── card.css            ← .card shell
    └── role-badge.css      ← .role-badge and modifiers

src/shared/components/
├── AppHeader.{tsx,css}
├── ConfirmDialog.{tsx,css}
├── ErrorBoundary.{tsx,css}
├── Spinner.{tsx,css}
└── ShortcutLegend.{tsx,css}

src/features/
├── auth/
│   ├── LoginPage.tsx
│   ├── RegisterPage.tsx
│   └── AuthPages.css       ← shared across login+register
└── tasks/
    ├── TaskPage.{tsx,css}
    ├── TaskDetailPage.{tsx,css}
    └── IntegrationIcon.{tsx,css}
```

**`src/index.css`** is the aggregator. It `@import`s tokens, typography, reset, and primitives in that order. Component files import their own CSS from their `.tsx`.

**Task ownership matrix** (when multiple agents work in parallel worktrees):

| Task type | Owns (edits freely) | Touches (read-only semantically) |
|-----------|---------------------|----------------------------------|
| `/colorize` | `tokens.css` | per-component: only swap hex → var() |
| `/typeset` | `typography.css` | per-component: font-family only |
| `/quieter` | per-component `.css` files | — |
| `/adapt` | `.tsx` files + new component `.css` | — |
| `/shape` | new component `.css` files | append-only on TaskPage.css |
| `/layout` | per-component `.css` files | tokens.css (rare) |

**Merge safety:** `.gitattributes` sets `*.css merge=union` as a fallback. If two tasks must touch the same file, union merge keeps both additions instead of producing conflict markers. The split-by-component approach means same-file edits should be the exception, not the rule.

**Rule:** never put component-specific styles in `tokens.css` / `typography.css` / `reset.css`. Never put global tokens in component files. When in doubt, grep where a selector is used — if it's in one component, it belongs in that component's file.
