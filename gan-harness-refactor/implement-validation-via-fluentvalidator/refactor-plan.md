# Refactor Plan — implement-validation-via-fluentvalidator

Track: backend
Baseline SHA: e21a3e403eeab550bed65b6c14db29fd41fdcd9e
Planner-version: 2 (amendment 2026-04-29: extends migration-parity exception to `validation_test_assertions`; adds axis 11; restates planned commits to reflect iter-1+iter-2 progress.)

## Goals

- Replace ad-hoc input validation (inline guard-throws and the static `FeatureValidation` helper) with idiomatic FluentValidation `AbstractValidator<T>` classes — one per gRPC request type — so request-shape rules live in named, testable, single-responsibility classes.
- Move the `RpcException` throw OUT of the validation step. Validators return `ValidationResult` only; the translation `ValidationResult → RpcException(InvalidArgument, ...)` happens in either the handler or a thin handler-wrapping decorator. After this refactor it is impossible for a `*Validator.cs` file to throw `RpcException` directly.
- Keep the externally observable error surface byte-identical: `StatusCode` and `Detail` strings (the human-readable validation messages today, e.g. `"feature_id is required"`, `"title too long"`, `"planned_end must be on or after planned_start"`) MUST be preserved. The behavior contract gates this.

- Axis 9 is a regression detector, not a coverage % gate. The SHARED auto-fail trigger ">2% coverage drop on touched files" is canonical; this axis just proves no previously-green test went red.

## Scope boundary

In scope:

- Introducing the `FluentValidation` (and `FluentValidation.DependencyInjectionExtensions`) NuGet packages on the three in-scope service `.csproj` files: `OneMoreTaskTracker.Users`, `OneMoreTaskTracker.Tasks`, `OneMoreTaskTracker.Features`.
- Authoring one `AbstractValidator<TRequest>` per validator-driven gRPC request type (≥ 8 mandatory). Each lives in its own file co-located with the request's handler subdirectory (e.g. `OneMoreTaskTracker.Features/Features/Create/CreateFeatureRequestValidator.cs`).
- Wiring the validators into DI with `services.AddValidatorsFromAssemblyContaining<TMarker>()` in each service's `Program.cs`.
- Choosing a `ValidationException → RpcException(InvalidArgument, ...)` translation strategy per service (decorator preferred for `Tasks` + `Features`, handler-prelude `ValidateAndThrowAsync` acceptable for `Users` given its single mega-handler) and implementing it once per service.
- Removing every inline `throw new RpcException(new Status(StatusCode.InvalidArgument, ...))` and every `FeatureValidation.*` call from handlers; deleting `OneMoreTaskTracker.Features/Features/Data/FeatureValidation.cs` once empty.
- Adding sibling `*ValidatorTests.cs` files that exercise each rule's pass + fail paths (AAA pattern, FluentAssertions; mirroring the existing test-organization convention).
- Reshaping any existing handler test that today provokes a guard-throw — those tests MUST still assert the same `RpcException` status code + Detail string post-refactor (no test rewrite that hides a behavior diff).

Out of scope (pinned for follow-up `/gan-refactor` runs, NOT silently absorbed):

- Translating non-validator `RpcException` paths (`NotFound`, `AlreadyExists`, `PermissionDenied`, optimistic-concurrency `AlreadyExists`) to a different mechanism. Those stay where they are; they are state-derived, not request-shape-derived.

## Feature-specific addenda

- **One type per file (project rule, 100%)**: every new validator class lives in its own file. No `Validators.cs` aggregator. No two `AbstractValidator<T>` declarations in one file. Axis 4 enforces.
- **Minimize comments (project rule, 100%)**: validator rules are self-documenting via fluent syntax; do NOT add `// validates that title is required` comments. The `WithMessage(...)` argument IS the documentation. Definitely no comments referencing iteration numbers, refactor task ids, or contract section numbers.
- **No log-only locals (project rule, 100%)**: validators do not log. Translators may log a single structured message at debug level when a validation failure is translated (`logger.LogDebug("validation failed: {Failures}", failures);`), but MUST NOT introduce intermediate locals just to feed the log statement.
- **Conventional Commits**: every commit in the planned sequence above uses `<type>(<scope>): <subject>`. Commit (a) is `chore`, (b)–(e) are `refactor`. Nothing breaking on the wire; no `!` suffix.
- **C# coding-style rules (`~/.claude/rules/csharp/coding-style.md`)**: validators are `public sealed class XxxValidator : AbstractValidator<XxxRequest>` (sealed; explicit access modifier). No `dynamic`. Rules registered in the constructor only. Nullable enabled (already on).
- **Microservice contract vocabulary (`~/.claude/rules/microservices/contracts.md`)**: validator class names live in their owning service's vocabulary. `Tasks` validators reference `JiraTaskId`/`AssigneeUserId` (already proto-correct), NEVER `UserId` bare. `Users` validators reference `Email`/`Password`/`ManagerId`. The Tasks-service `AttachTaskToFeatureRequestValidator` MUST NOT reach into Users-service concepts; the cross-service `UserId` field is already named `assignee_user_id`-equivalent on the wire.
- **Microservice security (`~/.claude/rules/microservices/security.md`)**: the `Register` validator MUST NOT introduce a settable `Role` rule for the self-registration path; today's handler hard-codes `Manager` for self-registration regardless of `request.Role`. The validator's role-membership check applies ONLY in the `When(r => r.ManagerId != 0)` branch (managed creation). This preserves the security invariant that `Role` is unsettable from public registration.
- **Decorator vs. handler-prelude split**: pinned per service in commit (a). Switching strategies mid-stream is allowed but requires a re-capture of the relevant `rpc_error_surface_*` surface to confirm no Detail-string drift; doing so is purely cosmetic and not recommended.
- **Request-shape vs state-driven Detail strings (axis 11, amendment 2026-04-29)**: every validator's `WithMessage(...)` argument MUST be authored from the perspective of "this rule failed because the request shape is invalid". A validator MUST NOT emit a Detail string whose baseline meaning is "this state could not be found / does not satisfy a runtime invariant". When a baseline state-driven message and a candidate request-shape message would be byte-identical, the cleaner resolution is to delete the validator rule entirely and let the existing state-driven check at the handler continue to emit it (preferred for `GetTeamRosterRequestValidator`'s `ManagerId > 0` rule).
